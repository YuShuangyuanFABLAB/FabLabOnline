# 代码变更追溯规范

## 核心问题

> **为什么问题反复出现，且每次都无法根据之前的经验快速修正？**
>
> **答案：缺乏改动追溯机制。**

---

## 一、问题反复出现的根本原因

### 1.1 改动无法追溯

每次修改代码时，没有系统性记录：

| 缺失信息 | 后果 |
|---------|------|
| 修改了什么 | 不知道哪些代码被改过 |
| 为什么修改 | 不知道修改的背景和目的 |
| 修改前状态 | 无法对比变化 |
| 验证结果 | 不知道修复是否真的有效 |
| 影响范围 | 不知道会影响哪些功能 |

### 1.2 症状与根因的映射缺失

当问题再次出现时：
- 不知道是否之前遇到过
- 不知道之前的解决方案是什么
- 不知道之前的方案是否完整
- 容易重复错误的修复路径

### 1.3 案例：边距问题的反复出现

```
第1次出现：边距变成0.5cm
  └── 修复：调整了某处代码
  └── 验证：手动看了一行文本，认为正确
  └── 记录：无
  └── 结果：问题根源未找到

第2次出现：边距又变成0.5cm
  └── 修复：又调整了某处代码
  └── 验证：手动看了几行文本，认为正确
  └── 记录：无
  └── 结果：问题根源仍未找到

第3次出现：边距随行数变化
  └── 分析：发现是公式计算问题
  └── 修复：使用COM API获取实际值
  └── 验证：测试3/5/8/10/13/15行，全部正确
  └── 记录：本次详细记录了
  └── 结果：找到根本原因
```

**前两次失败的原因**：
1. 没有详细记录修改内容
2. 没有验证边界情况
3. 没有建立问题与代码的映射关系

## 二、代码变更追溯规范

### 2.1 每次修改必须记录的信息

```markdown
## 变更记录：[日期] - [简短描述]

### 问题描述
- 症状：[用户看到的现象]
- 根因：[真正的原因，如果已知]

### 修改内容
- 文件：[文件路径]
- 位置：[函数名/行号]
- 修改前：[原代码或行为]
- 修改后：[新代码或行为]
- 原因：[为什么这样修改]

### 验证方法
- 测试场景：[测试了哪些情况]
- 测试结果：[每种情况的结果]
- 边界测试：[极端情况测试]

### 影响范围
- 可能影响的文件：[列表]
- 需要回归测试的功能：[列表]

### 相关问题
- 之前的相关修改：[链接到之前的记录]
- 相关skill：[链接]
```

### 2.2 变更记录的存放位置

**方案1：代码注释（适合小范围修改）**

```python
def _adjust_shape_height(self, shape):
    """
    调整形状高度

    变更记录：
    - 2026-02-20: 使用COM API获取实际文本高度
      - 原因：公式估算不准确，边距随行数变化
      - 验证：3/5/8/10/13/15行全部边距0.20cm
      - 参考：skill://ppt_layout_calculation
    """
```

**方案2：CHANGELOG文件（适合重要功能修改）**

```markdown
## [2026-02-20] 修复边距计算问题

### 问题
- 课堂知识内容边距随行数变化（5行0.31cm，13行0.60cm）

### 根因
- 形状高度使用公式估算，与PowerPoint实际渲染不一致

### 修改
- 新增 `adjust_shape_height_by_text()` 使用COM API获取实际文本高度
- 形状高度 = 实际文本高度 + 2 * 边距(0.4cm)

### 验证
- 测试3/5/8/10/13/15行，边距全部为0.20cm

### 参考
- skill: ppt_layout_calculation.md
```

**方案3：Git提交信息（必须包含关键信息）**

```
fix: 使用COM API修复边距计算问题

问题：边距随行数变化（公式估算不准确）
修复：使用TextRange.BoundHeight获取实际文本高度
验证：3/5/8/10/13/15行边距全部0.20cm
参考：.skills/ppt_layout_calculation.md
```

## 三、问题诊断快速路径

### 3.1 问题出现时的诊断步骤

```
Step 1: 检查是否有相关skill
        └── 读取相关skill文件
        └── 查看是否有类似问题的解决方案

Step 2: 检查Git历史
        └── git log --oneline --all --grep="关键词"
        └── git log -p --follow -- 文件路径
        └── 查看之前是如何修复的

Step 3: 检查代码注释
        └── 查看相关函数的变更记录注释
        └── 了解之前修改的原因

Step 4: 如果找到相关记录
        └── 分析之前修复是否完整
        └── 验证之前修复的代码是否被覆盖
        └── 检查是否有遗漏的修改点

Step 5: 如果没找到相关记录
        └── 创建新的问题诊断记录
        └── 确保这次详细记录
```

### 3.2 防止问题遗漏的检查清单

```
修复问题后：
□ 是否修改了所有相关位置？
  □ 搜索所有设置该属性的代码
  □ 确认是否有遗漏

□ 是否验证了边界情况？
  □ 最小值（0行、1行）
  □ 最大值（15行以上）
  □ 中间值（5、8、10行）

□ 是否更新了相关文档？
  □ skill文件
  □ 代码注释
  □ CHANGELOG

□ 是否考虑了影响范围？
  □ 其他使用该代码的地方
  □ 相关联的功能
```

## 四、经验教训

### 4.1 本次边距问题的教训

| 教训 | 说明 |
|------|------|
| 公式估算不可靠 | PowerPoint渲染复杂，要用实际值验证 |
| 多组测试必要 | 测试不同输入验证通用性 |
| 记录变更原因 | 便于后续追溯分析 |
| 验证影响范围 | 确保修复完整 |

### 4.2 通用原则

1. **修改前先记录状态** - 知道改了什么
2. **修改后验证多种情况** - 确保修复有效
3. **记录修改原因和验证结果** - 便于追溯
4. **更新相关skill** - 沉淀经验
5. **检查影响范围** - 避免遗漏
