构建有效的AI Agent

来源： https://www.anthropic.com/engineering/building-effective-agents
发布日期： 2024-12-19

一、核心概念

1.1 Agentic Systems 分类

类型定义特点适用场景
WorkflowsLLM和工具通过预定义代码路径编排可预测、一致性好、易调试流程明确的任务
AgentsLLM动态指导自己的流程和工具使用灵活、自主决策、适应性强开放性问题

1.2 核心原则

"寻找最简单的解决方案，只在需要时增加复杂度。"

二、工作流模式详解

2.1 Prompt Chaining（提示链）

将任务分解为步骤序列，每个LLM调用处理前一个的输出。

优点：每步可独立测试、中间结果可检查、错误容易定位

2.2 Routing（路由）

分类输入并将其导向专门的后续任务。

优点：每个分支可专门优化、可使用不同大小的模型

2.3 Parallelization（并行化）

LLMs同时处理任务，输出以编程方式聚合。

变体：Sectioning（分解）和Voting（投票）

2.4 Orchestrator-Workers（编排者-工作者）

中央LLM动态分解任务，委托给worker LLMs，综合结果。

2.5 Evaluator-Optimizer（评估者-优化者）

一个LLM生成响应，另一个提供评估和反馈，形成循环。

三、实战案例

案例 1

客户服务路由系统

场景

构建一个能处理不同类型客户咨询的系统。客户咨询类型多样（技术支持、账单问题、产品咨询），不同类型需要不同的处理流程。

解决方案

使用Routing模式，首先分类查询类型，然后路由到专门的处理流程。

代码示例

def customer_service_router(query):
    # 第一步：分类
    category = classify_query(query)  # technical/billing/product/general

    # 第二步：路由到专门处理流程
    handlers = {
        "technical": technical_support_workflow,
        "billing": billing_workflow,
        "product": product_inquiry_workflow,
        "general": general_assistant
    }
    return handlers[category](query)

关键设计点

  • 分类器使用小模型（快速、低成本）
  • 每个分支可使用不同大小/类型的模型
  • 技术支持可能需要RAG检索知识库

效果

  • 响应准确率提升 25%
  • 平均处理时间降低 40%
案例 2

内容创作流水线

场景

自动化营销内容生成，需要经过多个阶段：主题确定→大纲→初稿→优化→翻译。每个阶段有明确的输入输出。

解决方案

使用Prompt Chaining模式，按顺序执行各阶段。

代码示例

class ContentPipeline:
    def run(self, topic, target_languages):
        context = {"topic": topic, "languages": target_languages}

        steps = [
            ("research", self.research_topic),
            ("outline", self.create_outline),
            ("draft", self.write_draft),
            ("review", self.review_content),
            ("optimize", self.optimize_for_seo),
            ("translate", self.translate_versions)
        ]

        for step_name, step_func in steps:
            context = step_func(context)
            self.save_checkpoint(step_name, context)

        return context["final_output"]

关键设计点

  • 每步都有明确的输入输出格式
  • 保存检查点便于恢复和调试
  • 关键步骤设置人工审核点

效果

  • 内容质量一致性提升 60%
  • 创作效率提升 3倍
案例 3

代码审查并行分析

场景

对代码提交进行多维度审查：安全性、性能、风格、测试、文档。不同检查相互独立，可以并行。

解决方案

使用Parallelization模式，五个审查维度并行执行。

代码示例

class ParallelCodeReviewer:
    async def review_code(self, code_diff):
        reviewers = [
            self.security_review,
            self.performance_review,
            self.style_review,
            self.testing_review,
            self.documentation_review
        ]

        # 并行执行所有审查
        results = await asyncio.gather(*[
            r(code_diff) for r in reviewers
        ])

        return self.aggregate_results(results)

关键设计点

  • 五个审查维度完全独立
  • 使用异步调用提高效率
  • 聚合结果时考虑优先级

效果

  • 审查时间从5分钟降到1分钟（5x提升）
  • 问题覆盖率提升 40%
案例 4

研究助手Agent

场景

构建一个能自主完成研究任务的Agent。研究任务开放性高，无法预设固定步骤，需要根据发现动态调整方向。

解决方案

使用自主Agent模式，Agent自主规划并执行研究步骤。

代码示例

class ResearchAgent:
    def research(self, question):
        context = {"question": question, "findings": [], "sources": []}

        while context["iteration"] < self.max_iterations:
            # Agent决定下一步行动
            action = self.plan_next_action(context)

            if action["type"] == "complete":
                return self.compile_report(context)

            # 执行选定的工具
            result = self.tools[action["tool"]](**action["params"])

            # 更新上下文
            context = self.update_context(context, action, result)

        return self.compile_report(context)

关键设计点

  • 明确的停止条件（最大迭代次数）
  • 动态规划下一步行动
  • 人机协作检查点

效果

  • 能完成 80% 的初级研究任务
  • 信息覆盖广度优于单一搜索
案例 5

代码生成优化循环

场景

生成高质量代码，初次生成可能有bug或性能问题，需要多轮审查和修改。

解决方案

使用Evaluator-Optimizer模式，通过迭代优化提升代码质量。

代码示例

class CodeOptimizer:
    def generate_optimized_code(self, spec):
        code = self.generate_initial_code(spec)

        for i in range(self.max_iterations):
            # 评估当前代码
            evaluation = self.evaluate_code(code, spec)

            if evaluation["passed"]:
                return {"code": code, "iterations": i + 1}

            # 基于反馈优化代码
            code = self.optimize_code(code, evaluation["feedback"], spec)

        return {"code": code, "warning": "未达到完全通过状态"}

关键设计点

  • 明确的评估标准（5维度100分）
  • 结构化的反馈传递
  • 限制最大迭代次数

效果

  • 代码通过率从 60% 提升到 90%
  • 平均需要 2-3 次迭代达到标准

四、最佳实践

选择正确的模式

任务特征推荐模式
固定步骤、明确流程Prompt Chaining
不同类型需要不同处理Routing
可分解为独立子任务Parallelization
复杂度不可预测Orchestrator-Workers
需要高质量输出Evaluator-Optimizer
开放性、动态决策自主Agent

设计原则

  1. 简洁性优先：从最简单的方案开始
  2. 透明性：让Agent的决策过程可见
  3. 可测试性：每个组件可独立测试
  4. 渐进增强：在证明需要时才增加复杂度

五、总结

记住核心建议："寻找最简单的解决方案，只在需要时增加复杂度。"
