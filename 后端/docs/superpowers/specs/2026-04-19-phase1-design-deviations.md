## DEV-01: 密码登录范围扩展

| 项目 | 值 |
|------|-----|
| 设计文档章节 | 6.1 紧急登录通道 |
| 设计要求 | 密码登录**仅限 super_admin**，其他角色只能微信扫码 |
| 实际实现 | 所有用户均可密码登录 |
| 偏离原因 | 微信 OAuth 尚未上线（需 ICP 备案 + 微信开放平台审核），密码登录作为开发阶段临时方案 |
| 影响 | 普通用户在微信上线前可以使用密码登录 |
| 恢复计划 | 微信上线后，在 `auth.py` login 端点添加角色检查：非 super_admin 返回 403，引导至微信扫码 |
| 风险 | 低 — 后端 `require_permission` 已实现权限控制，密码登录范围扩大不影响数据安全 |

## DEV-03: 密码哈希算法 PBKDF2 替代 bcrypt

| 项目 | 值 |
|------|-----|
| 设计文档章节 | 6.1 紧急登录通道 |
| 设计要求 | 密码使用 bcrypt 存储 |
| 实际实现 | 使用 PBKDF2-SHA256（480,000 次迭代），自定义实现 |
| 偏离原因 | passlib/bcrypt 与 Python 3.14 不兼容 |
| 影响 | 安全性等效（PBKDF2-SHA256 480K 迭代 ≈ bcrypt cost 12） |
| 恢复计划 | Python 3.14 兼容性问题解决后评估迁移到 bcrypt 或 Argon2 |
| 风险 | 低 — PBKDF2 是 NIST 推荐算法，迭代次数足够 |

## DEV-04: JWT 传递方式 Cookie 替代 Bearer Token

| 项目 | 值 |
|------|-----|
| 设计文档章节 | 7.1 API 规范 |
| 设计要求 | JWT Token（Authorization: Bearer xxx） |
| 实际实现 | HttpOnly Cookie + SameSite=Strict |
| 偏离原因 | 安全审计（C1）要求：Cookie 防止 XSS 窃取 Token，比 Bearer Token 更安全 |
| 影响 | SDK 客户端不受影响（SDK 使用 keyring 存储 Token）；浏览器端安全性提升 |
| 恢复计划 | 不恢复 — 这是改进型偏离，Cookie 比 Bearer 更适合浏览器场景 |
| 风险 | 无 |

## DEV-05: 备份策略 pg_dump 替代云厂商自动备份

| 项目 | 值 |
|------|-----|
| 设计文档章节 | 12.1 备份与恢复 |
| 设计要求 | 云厂商自动备份（保留 30 天） |
| 实际实现 | pg_dump 脚本 + cron，保留 7 天日备 + 4 周周备 + 3 月月备 |
| 偏离原因 | 设计假设使用云数据库（RDS），实际使用 Docker 自建 PostgreSQL（成本考虑） |
| 影响 | 需要手动配置 cron；备份与数据库在同一 ECS（磁盘损坏风险） |
| 恢复计划 | 后续评估：(1) 阿里云 OSS 异地存储备份文件，或 (2) 迁移到云数据库 RDS |
| 风险 | 中 — 备份与数据库同机。缓解：脚本保留策略 + 定期恢复演练 |

## DEV-06: 监控方案简化

| 项目 | 值 |
|------|-----|
| 设计文档章节 | 12.2 监控与告警 |
| 设计要求 | 5 项指标告警：CPU > 80%、磁盘 > 85%、DB 连接数 > 80%、事件积压 > 1h、API 5xx > 5% |
| 实际实现 | `/health` + `/ready` + `/metrics` 端点；外部 UptimeRobot 健康检查；阿里云云监控 CPU/磁盘 |
| 偏离原因 | Phase 1 流量极低（<1 事件/秒），完整 Prometheus + Grafana 栈过度投入 |
| 影响 | 无法实时感知 DB 连接数、事件积压、API 错误率 |
| 恢复计划 | Phase 2 前部署 Prometheus + Grafana，配置设计文档要求的全部告警规则 |
| 风险 | 低 — 当前流量下问题可通过日志发现 |
